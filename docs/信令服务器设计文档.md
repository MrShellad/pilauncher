

# PiHub-Signal 信令服务器设计文档

## 1. 架构概述 (Overview)

PiHub-Signal 是专为 PiLauncher 设计的轻量级 WebRTC 信令服务器。其核心设计哲学是 **“社交归微软，通信归社区”**。服务器仅作为 P2P 打洞（SDP 与 ICE 交换）的透明转发层，不存储持久化的玩家社交关系，完全支持用户自建与社区联邦。

### 1.1 核心特性

* **去中心化与联邦机制**：支持官方节点、社区公共节点与玩家私有节点共存。
* **无感 Xbox 鉴权**：借助微软 XSTS Token 防止身份（XUID）伪造，无需自建账号系统。
* **极低资源占用**：纯 Rust (Tokio + Axum) 异步实现，单节点内存占用 < 50MB。
* **高可用/集群扩展**：支持基于 Redis Pub/Sub 的多节点水平扩展。

---

## 2. 系统拓扑拓扑 (Topology)

系统交互分为三层：

1. **社交身份层 (Xbox Live)**：PiLauncher 客户端直接与微软通信，获取 OAuth Token、好友列表及状态。
2. **信令握手层 (PiHub-Signal)**：负责校验玩家的 XSTS 票据，并通过 WebSocket 路由 Offer、Answer 和 ICE Candidate。
3. **数据传输层 (WebRTC P2P)**：打洞成功后，流量不再经过信令服务器，由两端 PiLauncher 底层直接通过 TCP-to-DataChannel 桥接 Minecraft 游戏流量。

---

## 3. 鉴权与防冒充机制 (Authentication)

为了防止恶意用户在社区节点上伪造他人的 XUID 拦截信令，握手阶段采用严格的票据校验机制：

1. **连接发起**：客户端发起 WebSocket 连接，首个数据包必须是 AuthRequest。
2. **票据提交**：包含客户端从微软获取的 XSTS Token 和 User Hash。
3. **服务端校验**：PiHub-Signal 后端带上该 Token，向 [https://xsts.auth.xboxlive.com](https://xsts.auth.xboxlive.com) 发起轻量级验证。
4. **身份绑定**：验证通过后，服务器从微软响应中提取真实的 XUID，并与当前的 WebSocket Session 唯一绑定。任何非该 XUID 的声明都将被服务器丢弃。

---

## 4. WebSocket 通信协议 (Protocol Definition)

所有信令交互均采用标准的 JSON 格式，包含外层的路由标识和内层的具体负载。

### 4.1 核心消息类型定义

| 消息类型 (type) | 方向 | 作用说明 | 关键载荷 (payload) |
| --- | --- | --- | --- |
| auth_req | C -> S | 客户端请求鉴权 | xsts_token, user_hash |
| auth_res | S -> C | 服务端返回鉴权结果 | status (ok/error), xuid |
| rtc_offer | C -> S -> C | 房主向访客发送打洞邀请 | target_xuid, sdp (字符串) |
| rtc_answer | C -> S -> C | 访客同意打洞并回复 | target_xuid, sdp (字符串) |
| ice_candidate | C -> S -> C | 交换网络候选路径以打洞 | target_xuid, candidate_data |
| presence_sync | C -> S -> C | 广播当前游戏富状态 | version, modpack, dimension |

*注：C 代表 Client (PiLauncher)，S 代表 Server (PiHub-Signal)。*

## 5. 跨服寻址与联邦设计 (Federation & Routing)

为了打破不同信令服务器之间的“孤岛效应”，系统引入 **PiHub URI 契约**。

### 5.1 房间链接机制

当房主（连在 server-A.com）开启局域网时，生成如下标准 URI：
pihub://[server-A.com/join?xuid=2535421111111111](https://server-A.com/join?xuid=2535421111111111)

### 5.2 跨服直连流程

1. 访客（目前连在 server-B.com）在 Ore UI 界面点击加入该房间，或解析该 URI。
2. 访客的 PiLauncher 识别到目标信令服务器为 server-A.com，判断与当前节点不一致。
3. 访客的 Rust 后端在后台静默建立一个通往 server-A.com 的临时 WebSocket 连接。
4. 在 server-A.com 上完成 Auth 鉴权并与房主完成 WebRTC SDP 交换。
5. WebRTC 物理连接建立后，安全销毁临时的 WebSocket 连接。

## 6. 容灾与高可用架构 (High Availability)

为应对大规模突发流量或单点故障，官方与大型社区节点可开启集群模式：

### 6.1 Redis Pub/Sub 路由

* 所有 PiHub-Signal 节点配置相同的 Redis 集群连接。
* 当玩家连入节点 A，节点 A 向 Redis 订阅频道 channel:xuid:<该玩家XUID>。
* 若节点 B 收到发往该玩家的信令，直接 PUBLISH 到该频道，节点 A 接收后通过 WebSocket 下发，实现无状态的跨节点路由。

### 6.2 客户端自愈与退避策略

* **心跳保活**：客户端每隔 20 秒发送 ping，若 10 秒无 pong 响应则判定断线。
* **指数退避机制**：断线后，重连间隔按 1s -> 2s -> 4s -> 8s -> 16s 递增（并附带 ±20% 随机抖动），防止服务器重启瞬间遭遇连接雪崩。
* **降级目录**：客户端可订阅统一的 servers.json 目录，主节点宕机后，自动 fallback 至延迟最低的健康备用节点。

---

## 7. 部署与打包规范 (Deployment)

1. **统一分发**：提供预编译的单文件可执行程序（Linux/Windows/macOS）以及轻量级 Docker 镜像。
2. **环境变量配置**：
* PIHUB_PORT: 监听端口（默认 8080）。
* PIHUB_REDIS_URL: 若配置，则自动开启集群模式；若留空，则作为独立私有节点运行。
* PIHUB_ALLOW_UNSAFE: 开发者模式开关，开启后可跳过 XSTS 严格校验，便于局域网内网测试。

---

## 8. NAT 穿透兜底与 TURN 集成 (NAT & Relay)

在国内复杂的网络环境下（如对称型 NAT 叠加多层大内网），纯粹的 STUN 打洞成功率无法达到 100%。当 WebRTC 无法建立 P2P 直连时，必须依靠 TURN 服务器进行流量中转。

### 8.1 动态凭证下发 (Ephemeral TURN Credentials)

PiHub-Signal 不负责实际的中继流量（避免高昂的带宽成本），但它需要负责**下发中继服务器的加密凭证**：

* 当客户端通过 auth_req 鉴权成功后，服务端会在 auth_res 中附带一组由 HMAC-SHA1 动态签名的临时 TURN 账号密码（基于 RFC 5389 的 Time-Limited 机制）。
* 该凭证仅在接下来 24 小时内有效，且与玩家的 XUID 强绑定，防止中继带宽被黑客盗刷。

```json
// auth_res 载荷扩充
{
  "type": "auth_res",
  "status": "ok",
  "xuid": "2535421111111111",
  "turn_config": {
    "uris": ["turn:turn.community.org:3478?transport=udp"],
    "username": "1684713600:2535421111111111",
    "credential": "base64_encoded_hmac_signature"
  }
}

```

## 9. 安全管控与 QoS 限流 (Security & QoS)

为了防止社区公开节点被恶意脚本打挂（例如疯狂发送垃圾信令导致内存溢出 OOM），系统需要在网关层和应用层实现双重保护：

### 9.1 连接与负载限制

* **Max Payload Size**：严格限制单个 WebSocket 帧的最大体积（例如 16KB）。WebRTC 的 SDP 和 ICE 数据通常极小，任何超过此阈值的消息直接断开连接。
* **连接数硬顶 (Max Connections)**：配置单节点最大并发连接数（如 10,000）。达到上限后，健康检查接口返回 503 Service Unavailable，引导负载均衡器将流量打向备用节点。

### 9.2 令牌桶限流 (Token Bucket Rate Limiting)

* **信令风暴拦截**：基于 XUID 级别在内存中维护令牌桶。限制每个玩家每秒最多发送 10 条信令。如果突发频率超过阈值（如恶意狂发 rtc_offer），服务端自动丢弃多余包，并下发 rate_limited 警告信号。

---

## 10. 可观测性与遥测 (Observability)

作为系统管理员或社区服主，需要实时掌握服务器的健康状态。PiHub-Signal 原生集成监控导出能力。

### 10.1 Prometheus Metrics 导出

通过单独开启一个不暴露在公网的管理端口（如 9090），提供 /metrics 接口，暴露以下关键指标供 Grafana 抓取：

* pihub_connected_clients：当前在线的长连接玩家总数。
* pihub_messages_routed_total：累计路由的信令条数（用于评估系统吞吐）。
* pihub_auth_failures_total：Xbox XSTS 鉴权失败的次数（如果突然飙升，说明可能有黑客在尝试撞库或伪造身份）。
* pihub_memory_usage_bytes：Rust 进程当前内存占用。

## 11. 客户端桥接上下文 (Client-Side Proxy Context)

*注意：此部分说明客户端 PiLauncher 如何配合信令服务器完成最终的闭环。*

1. **房主端 (Host)**：游戏在本地端口（如 25565）开启局域网后，PiLauncher 的 Rust 后端创建一个 TCP Server 持续读取该端口数据，并通过已建立的 WebRTC DataChannel 发送出去。
2. **访客端 (Guest)**：打洞成功后，PiLauncher 的 Rust 后端随机映射一个本地端口（如 127.0.0.1:30000）。访客在 Minecraft 的多人游戏中添加此本地地址，连接产生的 TCP 流量将被全量塞入 DataChannel，跨过公网送达房主端。
