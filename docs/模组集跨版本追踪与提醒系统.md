# 模组集跨版本追踪与提醒系统

## 1. 目标

用户可以把一组相关 Mod 保存为一个“模组集”，例如“农夫乐事全家桶”“科技基础套件”。系统以项目 ID 为核心追踪这些 Mod 是否已经支持目标环境，例如 Minecraft `1.21.2` + `Fabric`，并在集合达到可迁移状态时提醒用户。

这个系统的关键不是“收藏某个 jar 文件”，而是“收藏项目，并追踪它在目标版本上的可用版本”。

## 2. 范围

### 2.1 首期范围

- 从资源下载页或实例内下载页的多选结果创建 / 加入模组集。
- 为一个模组集开启目标环境追踪：游戏版本 + Loader。
- 手动刷新或启动器后台静默检查就绪度。
- 在资源库展示追踪进度卡片。
- 就绪度达到 100% 时显示启动器内提醒。
- 支持一键进入“查看详情”，展示每个 Mod 的状态与匹配到的最新文件。

### 2.2 暂不做

- 自动升级实例。
- 自动替换当前实例里的旧 Mod。
- 云同步跨设备提醒。
- 复杂依赖图求解。

这些能力可以在追踪结果稳定后作为第二阶段加入。

## 3. 核心概念

### 3.1 模组集

模组集是 `collections` 的一种类型，建议新增类型值：

```text
collection.type = "mod_set"
```

模组集保存的是项目引用，不保存某个具体文件。每个成员来自 `starred_items`，通过 `collection_items` 建立关系。

### 3.2 目标环境

目标环境由两个字段组成：

```text
game_version = "1.21.2"
loader = "fabric" | "forge" | "neoforge" | "quilt"
```

资源包和光影未来也可以追踪，但首期建议只追踪 Mod，因为 Loader 适配是主要痛点。

### 3.3 就绪度

就绪度是集合中可用于目标环境的项目数量占比：

```text
ready_count / total_count
```

状态分为：

- `ready`：找到匹配目标版本和 Loader 的可下载文件。
- `pending`：项目存在，但尚未发布目标环境版本。
- `unknown`：检查失败、项目不可访问或接口异常。
- `ignored`：用户手动忽略，不计入总数。
- `removed`：项目已下架或无法解析。

## 4. 数据模型

现有表：

- `starred_items`：收藏的项目快照。
- `collections`：收藏集。
- `collection_items`：收藏集与项目关系。

建议新增三张表。

### 4.1 `mod_set_trackers`

一个模组集可以有多个追踪目标，例如同时追踪 `1.21.2 Fabric` 和 `1.21.4 NeoForge`。

```sql
CREATE TABLE IF NOT EXISTS mod_set_trackers (
  id TEXT PRIMARY KEY,
  collection_id TEXT NOT NULL,
  game_version TEXT NOT NULL,
  loader TEXT NOT NULL,
  status TEXT NOT NULL DEFAULT 'active',
  readiness_status TEXT NOT NULL DEFAULT 'checking',
  ready_count INTEGER NOT NULL DEFAULT 0,
  total_count INTEGER NOT NULL DEFAULT 0,
  last_checked_at INTEGER,
  next_check_at INTEGER,
  notified_ready_at INTEGER,
  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL
);
```

字段说明：

- `status`: `active | paused | archived`
- `readiness_status`: `checking | partial | ready | error`
- `notified_ready_at`: 防止 100% 后重复打扰用户。

### 4.2 `mod_set_tracker_items`

保存每个项目在目标环境下的检查结果。

```sql
CREATE TABLE IF NOT EXISTS mod_set_tracker_items (
  id TEXT PRIMARY KEY,
  tracker_id TEXT NOT NULL,
  item_id TEXT NOT NULL,
  source TEXT NOT NULL,
  project_id TEXT NOT NULL,
  status TEXT NOT NULL,
  matched_version_id TEXT,
  matched_version_number TEXT,
  matched_file_name TEXT,
  matched_download_url TEXT,
  published_at TEXT,
  checked_at INTEGER,
  error TEXT,
  extra TEXT,
  UNIQUE(tracker_id, item_id)
);
```

`extra` 用于保存 API 原始摘要、依赖数量、是否 beta/alpha 等可选信息。

### 4.3 `mod_set_notifications`

记录提醒事件，方便首页、资源库和通知中心读取。

```sql
CREATE TABLE IF NOT EXISTS mod_set_notifications (
  id TEXT PRIMARY KEY,
  tracker_id TEXT NOT NULL,
  type TEXT NOT NULL,
  title TEXT NOT NULL,
  body TEXT NOT NULL,
  read_at INTEGER,
  created_at INTEGER NOT NULL
);
```

`type` 首期可用：

- `tracker_ready`
- `tracker_progress`
- `tracker_error`

## 6. 检查算法

输入：

- `collection_id`
- `game_version`
- `loader`

流程：

1. 读取 `collection_items`，解析对应 `starred_items`。
2. 对每个项目按 `source` 分派：
   - Modrinth: `get_ore_project_versions(project_id, game_version, loader)`
   - CurseForge: `fetchCurseForgeVersions(project_id, game_version, loader)`
3. 过滤候选版本：
   - `game_versions` 包含目标游戏版本。
   - `loaders` 包含目标 Loader。
   - 文件存在 `download_url` 和 `file_name`。
4. 选择最新版：
   - 优先按 `date_published` 降序。
   - 同日按 `version_number` 字符串兜底排序。
5. 写入 `mod_set_tracker_items`。
6. 计算 `ready_count`、`total_count`、`readiness_status`。
7. 若从非 100% 变为 100%，写入通知，并设置 `notified_ready_at`。

伪代码：

```ts
for item in collectionItems:
  versions = await fetchVersions(item.source, item.projectId, targetVersion, loader)
  match = versions
    .filter(canRunOnTarget)
    .sort(byPublishedDesc)[0]

  saveResult(item, match ? "ready" : "pending", match)

tracker.readyCount = results.count(status === "ready")
tracker.totalCount = results.count(status !== "ignored")
tracker.status = readyCount === totalCount ? "ready" : "partial"
```

## 7. 轮询策略

### 7.1 触发时机

- 用户手动点击“检查更新”。
- 启动器启动后延迟检查。
- 下载页 / 资源库打开时，如果上次检查超过 6 小时则刷新。

### 7.2 节流

建议默认间隔：

```text
active tracker: 6 小时
已 100% ready: 24 小时
error tracker: 30 分钟后重试，最多退避到 12 小时
```

### 7.3 并发控制

- 同一 tracker 同一时间只允许一个检查任务。
- 单 tracker 内项目并发限制为 4。
- 全局 API 并发限制为 6。
- 失败项目单独标记 `unknown`，不让整个 tracker 失败。

## 8. UI/UX 设计

### 8.1 创建模组集

入口：

- 下载页多选后的底部 Action Bar。
- 实例内下载页多选后的底部 Action Bar。
- 资源库页面的“新建收藏集”。

弹窗内容：

- 集合名称。
- 集合描述，可选。
- 已选择 Mod 列表。
- 是否立即开启版本追踪。

如果开启追踪，则显示：

- 目标 Minecraft 版本。
- Loader 类型。
- 初次检查按钮。

### 8.2 资源库进度卡片

卡片内容：

```text
农夫乐事全家桶
目标：Minecraft 1.21.2 / Fabric
进度：8 / 12 已就绪
状态：等待 4 个项目更新
```

视觉建议：

- 使用 Ore UI 的 2px 黑色边框。
- Ready 项使用绿色状态条。
- Pending 项使用黄色状态条。
- Unknown 项使用红色小徽标。
- 100% 后卡片升级为绿色高亮状态。

操作：

- `查看详情`
- `立即检查`
- `一键下载到实例`
- `暂停追踪`

### 8.3 追踪详情

详情列表字段：

- 图标、名称、作者。

- 匹配到的版本号。

- 文件名。

### 8.4 提醒

提醒位置：


## 9. 状态机

### 9.1 Tracker 状态

```text
active -> checking -> partial
active -> checking -> ready
active -> checking -> error
ready -> checking -> partial  // 如果某项目被移除或筛选规则变化
active -> paused
paused -> active
active -> archived
```

### 9.2 Item 状态

```text
pending -> ready
pending -> unknown
unknown -> ready
unknown -> pending
ready -> pending     // 目标规则变化后重新计算
pending -> ignored
ready -> ignored
```

## 10. 与现有系统的关系

### 10.1 Library

模组集继续走现有 Library 数据结构：

- `collections`: 保存集合本体。
- `collection_items`: 保存项目成员。
- `starred_items`: 保存项目快照。

新增 tracker 只负责“目标环境追踪”，不替代收藏系统。

### 10.2 下载页多选

当前下载页已经有多选 Action Bar，可以把“加入收藏夹”升级为：

```text
加入收藏夹 -> ModSetCreateModal
```

用户可以选择：

- 加入已有模组集。
- 新建模组集。
- 新建并开启追踪。

### 10.3 下载任务

当 tracker ready 后，“一键下载”应复用现有下载管线：

- `download_resource`
- `modService.updateModManifest`
- `DownloadManager`

不要单独实现下载器。

## 11. 边界情况

- 项目跨平台 ID 不一致：同一个 Modrinth / CurseForge 项目先按来源独立追踪，后续再做合并映射。
- Loader 名称差异：统一小写标准值，如 `neoforge`。
- 项目只发布 alpha/beta：首期默认接受所有 release type，后续加入稳定版过滤。
- 某 Mod 不再维护：用户可以标记 ignored。
- API 失败：保留上次成功结果，并把本次错误写入 `error`。
- 集合为空：不允许开启追踪。
- 目标版本为空：不允许创建 tracker。

## 12. 分阶段落地

### Phase 1: 收藏与本地追踪

- 替换收藏占位弹窗为 `ModSetCreateModal`。
- 新增 tracker 数据表与命令。
- 支持手动检查。
- 资源库展示进度卡片。

### Phase 2: 静默检查与提醒

- 启动器启动后后台检查。
- 写入 `mod_set_notifications`。
- 首页 / 资源库展示提醒。
- 100% 后防重复提醒。

### Phase 3: 一键部署

- 详情页展示 ready 项。
- 批量下载到指定实例。
- 对当前实例做已安装检测。
- 支持“只下载缺失项目”。

### Phase 4: 高级能力

- 稳定版过滤。
- 依赖关系提示。
- 多目标版本追踪。
- 跨源项目映射。
- 云同步。

## 13. 验收标准

- 用户可以从多选资源创建一个模组集。
- 用户可以为模组集设置目标版本和 Loader。
- 系统能计算并展示 `ready_count / total_count`。
- 对于 ready 项，系统能记录匹配文件名、版本号和下载链接。
- 进度达到 100% 时只提醒一次。
- API 失败不会清空已有成功结果。
- 一键下载复用现有下载任务队列。
