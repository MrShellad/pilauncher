# 模组更新检查机制分析报告与优化建议

## 1. 现状概述
当前的模组更新检查逻辑主要实现在 `src/features/InstanceDetail/hooks/useModManager.ts` 中的 `runUpdateCheck` 函数。该机制通过轮询实例中的模组列表，逐一调用 Modrinth API 获取最新版本，并对比本地 `fileId` 与云端 `latest.id` 来确定是否存在更新。

## 2. 核心问题分析

### 2.1 性能与响应速度
*   **串行轮询瓶颈**：当前采用 `for...of` 循环配合 `setTimeout(resolve, 200)` 的方式。对于拥有 200 个模组的整合包，仅等待延迟就需要 40 秒，极大地降低了用户获取更新信息的及时性。
*   **未利用批量 API**：Modrinth 支持通过多个项目 ID 一次性获取数据，但目前采用的是一模一请求的模式，增加了网络开销和后端负载。

### 2.2 渲染性能
*   **高频状态触发**：每检查完一个模组即调用 `setMods` 触发全量列表更新。在大型列表中，这会导致 React 进行大量无效的 Diff 计算，引发界面微卡顿。

### 2.3 逻辑可靠性
*   **单一匹配风险**：完全依赖 `source.fileId` 进行匹配。对于手动导入、元数据损坏或非 Modrinth 来源转 Modrinth 匹配的模组，该机制往往失效。
*   **缺乏版本权重判断**：仅对比 ID 是否一致，无法处理版本撤回、跨频道（Release/Beta）更新等复杂场景。

### 2.4 资源管理
*   **异步泄露隐患**：`runUpdateCheck` 循环缺乏取消机制。当用户离开实例详情页或切换实例时，之前的更新循环仍在后台运行，浪费系统资源。
*   **缓存覆盖策略**：每次检查都会创建新的 Map 并覆盖旧缓存，导致增量检查无法实现。

---

## 3. 优化建议

### 3.1 性能优化
*   **[建议] 并发控制**：引入并发池（控制同时运行的请求数为 5-8 个），或者在 Rust 后端实现批量请求逻辑。
*   **[建议] 状态更新分片**：将 `setMods` 的频率降低。例如，每 2 秒更新一次结果，或者每完成 10% 的进度更新一次。

### 3.2 逻辑增强
*   **[建议] 引入哈希校验**：当 `fileId` 缺失时，通过文件的哈希值向 Modrinth 查询版本，提高手动导入模组的更新覆盖率。
*   **[建议] 版本号对比**：引入语义化版本（SemVer）对比库，确保“更新”是指向更高版本，而不是仅仅指向不同的 ID。
*   **[建议] 频道选择**：在设置中允许用户选择“稳定版”或“预览版”频道。

### 3.3 稳定性保障
*   **[建议] 中断信号**：使用 `AbortController` 传递信号到更新循环，确保组件卸载时立即停止所有待处理请求。
*   **[建议] 增量更新缓存**：修改缓存逻辑，采用 `merge` 策略。如果某个模组在本次会话中已检查过且未过期，则直接跳过。

## 4. 实施优先级
1.  **高**：分片状态更新（解决 UI 卡顿）。
2.  **高**：并发请求处理（解决检查缓慢）。
3.  **中**：引入 `AbortController`（解决资源占用）。
4.  **低**：基于哈希的备选匹配（解决兼容性）。
