
# 📄 设计文档：第三方实例导入与全局资源复用

## 1. 业务背景与目标
* **背景：** 玩家在迁移至新启动器时，通常在本地已有庞大的 `.minecraft` 目录（来自 PCL 或 HMCL）。要求玩家重新下载数百兆的 `assets` 和 `libraries` 会严重阻碍转化率。
* **目标：**
  1. 支持一键扫描或手动选择第三方启动器的游戏目录。
  2. 无损提取并转换隔离的 `versions/{instance_name}` 为本启动器的标准实例模型。
  3. **核心目标：** 实现 `assets` 和 `libraries` 的“零拷贝”复用，极大地节省玩家磁盘空间。

## 2. 资源复用策略：硬链接 (Hard Links) / 软链接 (Symlinks)
为了在你的 `runtime/assets` 和 `runtime/libraries` 中复用第三方文件而不占用双倍空间，最优雅的底层方案是使用**硬链接 (Hard Links)**，辅以**复制 (Copy)** 作为跨盘符的降级方案。

* **为什么不用软链接？** 软链接（快捷方式）在 Windows 上有时需要管理员权限，且如果源文件被删除，软链接会失效。
* **硬链接的优势：** 多个文件路径指向磁盘上的同一块物理数据。PCL 和本启动器都可以安全读取。即使玩家卸载了 PCL 并删除了 PCL 的目录，只要本启动器的硬链接还在，物理文件就不会丢失。

## 3. 核心架构设计

导入模块可拆分为三个核心子系统：

### 3.1 探测与识别器 (Scanner)
* **输入：** 用户选择的一个目标 `.minecraft` 文件夹路径。
* **逻辑：**
  * 校验目录结构：检查是否存在 `assets`, `libraries`, `versions` 子目录。
  * 遍历 `versions/` 目录，通过识别特征文件（如 `hmcl.json`, `pcl.json`, 包含 `mods/` 文件夹等）过滤出“已隔离的实例”和“原版核心”。
* **输出：** 一个可供导入的实例列表模型，供前端 UI 渲染。

### 3.2 资源池链接器 (Resource Linker)
负责处理 `assets` 和 `libraries` 的复用逻辑。
* **文件比对：** 提取源目录和目标目录的文件哈希（或简单依靠文件大小+最后修改时间）进行对比。如果你的 `runtime/` 中已经存在完全相同的文件，则跳过。
* **建立链接：**
  1. 遍历第三方 `libraries/` 和 `assets/objects/` 下的所有文件。
  2. 在你的 `runtime/libraries/` 和 `runtime/assets/objects/` 中创建对应的目录结构。
  3. 执行 `fs::hard_link(src_file, dest_file)`（Tauri 后端执行）。
  4. **降级处理：** 捕获操作系统错误（例如 `EXDEV`: 源文件和目标文件不在同一个逻辑磁盘分区）。如果跨盘，则自动降级为普通的文件拷贝 (Copy)。

### 3.3 实例转换器 (Instance Converter)
将第三方实例接入你现有的 `instances/` 体系，或者保持在原处但纳入你的管理。

* **方案 A（外部挂载模式）：** 不移动原文件夹，在你的系统中创建一个虚拟配置，其 `GameDir` 指向原 PCL 目录。

## 4. 详细交互与执行流程

### 阶段一：前端交互 (React 端)
1. **触发导入：** 用户在 UI 点击“导入本地游戏”。
2. **选择目录：** 唤起原生文件选择器，用户选择旧版 `.minecraft` 目录。
3. **展示结果：** 前端调用后端扫描接口，拿到实例列表，在界面上渲染（例如：“发现 1.16.5-RLCraft，1.20.1-Fabric 等 5 个实例”）。
4. **配置选项：** 提供一个 Checkbox：`[x] 尝试通过硬链接复用依赖文件 (极速且节省空间)`。
5. **执行进度：** 点击“开始导入”，弹出进度条界面（监听后端发送的事件：`Linking Libraries... 45%`, `Copying Instance...`）。

### 阶段二：后端执行 (Tauri / Rust 端)
后端接收到导入指令和配置后，执行以下 Pipeline：

```text
[Pipeline 开始]

1. 锁定环境 (加锁避免同时启动游戏导致冲突)

2. 执行 Resource Linker (如果勾选了复用)
   -> 获取 src: D:\PCL\.minecraft\libraries
   -> 获取 dest: D:\PiLauncher\runtime\libraries
   -> 并发遍历源文件 (使用类似 rayon 库提升性能)
   -> 尝试 Hard Link -> 失败则 Copy -> 记录成功/失败日志。
   -> 同理处理 assets/ 目录。

3. 执行 Instance Converter
   -> 获取选定的 versions/xxx 目录
   -> 复制该目录到 PiLauncher 的 instances/xxx 
   -> (关键) 删除 instances/xxx 下的 .jar 和基础 .json（因为你的架构核心放在 runtime/versions/ 中）
   -> 解析原 json，提取 "inheritsFrom" (如 1.20.1)，确保该核心在你的 runtime/versions 中已注册或已链接过来。
   -> 生成你自己的 instance_config.json

4. 广播完成事件到前端。

[Pipeline 结束]
```

## 5. 异常处理与边界情况

1. **跨盘符硬链接失败：**
   * **场景：** 玩家的 PCL 在 `C:` 盘，新启动器在 `D:` 盘。
   * **处理：** 拦截硬链接错误码，静默转为普通的异步大文件拷贝。在 UI 结束时可以给个提示：“由于源文件和启动器不在同一磁盘，已降级为普通复制，未节约空间”。
2. **文件占用/权限问题：**
   * **场景：** 玩家后台正开着旧启动器或游戏，导致某些文件被锁定。
   * **处理：** 导入前进行目录级别的写入测试，若被占用，立刻中断并提示“请先关闭其他启动器和游戏”。
3. **Assets Index 缺失：**
   * **场景：** 某些极端情况下，旧启动器的 `assets/indexes/` 下的 json 损坏。
   * **处理：** 将 `assets/objects/` 的文件尽可能链接过来，但在首次启动该实例时，触发一次常规的完整性校验修复（Asset Checker）。

## 6. 核心优势总结
采用**硬链接资源复用 + 独立实例存储**的设计，完美契合了现代分离式架构。玩家导入 50G 的整合包目录，可能只需要几秒钟的时间（因为建立文件系统的链接极快，无需拷贝数据本身），这将在用户体验上形成巨大的“Wow Moment”。



尤其是结合你的后端底层技术，这几个点能极大提升程序的健壮性：

### 1. 区分“只读数据”与“读写数据”的处理策略
前面我们提到用**硬链接（Hard Link）**来处理 `assets` 和 `libraries`，这是因为它们是**只读的静态资源**。但对于游戏内的其他文件夹，策略必须不同：

* **绝对不能硬链接的目录：** `saves/`（世界存档）、`logs/`、`crash-reports/`。
    * **原因：** 如果你把存档硬链接过来，玩家在你的启动器里玩了一会儿，不小心又打开 PCL 玩了同一个档，极容易造成区块数据损坏或进程文件锁冲突。
    * **策略：** 这些目录必须使用**深度拷贝（Deep Copy）**。
* **视情况而定的目录：** `mods/`、`config/`。
    * **策略：** 同样推荐**直接拷贝**。如果玩家后续在你的启动器里删除了某个 Mod，你肯定不希望 PCL 那边的 Mod 也跟着神秘消失（因为硬链接是双向的物理文件映射）。

### 2. 导入事务与回滚机制 (Rollback)
由于导入整合包可能涉及几千个文件的拷贝和链接，一旦玩家中途点击“取消”，或者遇到磁盘满、权限不足等错误，决不能留下一个“半残缺”的实例在你的 `instances/` 目录里。

* **事务逻辑 (Transaction-like)：**
    1.  在 Rust 后端创建一个临时的导入目录，例如 `instances/.temp_importing_xxx/`。
    2.  所有的拷贝、链接操作都在这个临时目录中进行。
    3.  如果遇到 `Error` 或者玩家触发 `Cancel`，直接调用 `fs::remove_dir_all` 销毁临时目录，保证主界面的干净。
    4.  只有当所有文件操作100%成功，并且生成了最终的 JSON 配置文件后，才执行最后一步：将目录重命名为正式的 `instances/xxx/`，并通知前端刷新列表。

### 3. 跨盘导入的性能极限优化
如果遇到无法建立硬链接（跨盘符）必须降级为普通拷贝的情况，大量小文件（比如 `assets/objects` 里的几千个哈希文件）拷贝会非常慢。

* **并发拷贝：** 既然你的底层拥有极高的性能上限，不要用单线程的 `for` 循环去拷贝文件。对于大量的碎文件，可以使用类似 `rayon` 这样的并行计算库，或者利用 `tokio` 生成一批异步的 I/O 任务。将 `assets` 文件夹的拷贝速度压榨到磁盘 I/O 的极限，这能成为你启动器的一大卖点。
