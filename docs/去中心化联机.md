# 📜 MC P2P 联机核心引擎 (Core Engine) 详细设计文档 V1.1 (Sidecar 架构版)

## 1. 系统定位与设计哲学

本系统（命名代号：`NetCore-Sidecar`）是一个由 Rust 编写的、无 UI 的极简伴生进程。它的唯一职责是：**为上层应用（Minecraft 启动器）提供极其简单、合法合规、且极具穿透力的底层网络直连隧道。**

### 1.1 核心原则
* **Sidecar 伴生模式 (同生共死)**：引擎没有自己的独立生命周期，完全作为主程序（启动器）的附属进程运行。当启动器关闭、游戏结束或系统管道断开时，引擎自动优雅销毁，绝不在后台残留。
* **零端口冲突 IPC**：废弃传统的本地 WebSocket/HTTP 监听，全面改用 OS 级别的 **Stdio (标准输入/输出)** 进行 JSON-RPC 通信，实现物理级隔离与极速交互。
* **中国网络特化**：绝对优先 IPv6 直连；采用 UDP + TCP 双引擎打洞对抗运营商 QoS 限速；内置私有化 TURN 中继回退机制。
* **合规与安全边界**：强制端到端加密（E2EE）；**彻底阉割**节点间的非受信任中继路由功能（杜绝 PCDN 滥用）；官方仅提供开源服务端，鼓励玩家/社区私有化部署。

## 2. 系统整体架构

整个联机生态被严格划分为三个物理/逻辑层：

### 2.1 表现与控制层 (Launchers / UI)
* **包含**：PiLauncher、HMCL、PCL2 等第三方启动器，或玩家自制的极简 UI。
* **职责**：管理游戏版本、启动参数、Mods；向玩家展示生成的“房间邀请码”；通过系统进程 API 拉起引擎；通过标准输入流（Stdin）发送控制指令；渲染网络延迟/状态 UI。

### 2.2 核心引擎层 (Rust NetCore-Sidecar)
* **包含**：作为附属二进制文件打包在启动器内部的跨平台无 UI 进程。
* **职责**：高频监听标准输入（Stdin）；连接信令服务器进行 SDP/ICE 候选者交换；执行并发打洞（Hole Punching）；在本地按需闪现 TCP 伪装端口；截获并封装游戏流量。

### 2.3 信令与中继层 (Signaling & TURN Server)
* **包含**：基于轻量级 WebSocket 的信令服务和标准 TURN 服务（可通过 Docker 一键部署）。
* **职责**：仅负责握手期的元数据交换；在打洞彻底失败时，提供纯粹的加密字节流盲转发。

## 3. 核心技术流与模块设计

### 3.1 流量劫持与隧道复用 (Tunneling Module)
此模块负责将 Minecraft 原生的 TCP 协议“塞进”去中心化的 P2P 隧道中。

* **底层协议栈**：采用 `webrtc-rs` 的核心组件（SCTP over DTLS over UDP/TCP）。
* **多路复用机制**：
  1. 房主端与客户端一旦通过 ICE 打洞成功，即建立一条底层的 `PeerConnection`。
  2. 客户端引擎收到隧道就绪信号后，瞬间在本地暴露一个 TCP 代理端口（如 `127.0.0.1:50001`）。
  3. 客户端启动器拉起 MC 并连接该端口，客户端引擎的 `tokio` 监听到 TCP 连接，立即在 `PeerConnection` 上开启一个全新的 `DataChannel` (可靠、有序模式，完美模拟 TCP)。
  4. 房主端引擎收到 `OnDataChannel` 事件后，立即向本地的 MC 服务端 `127.0.0.1:25565` 发起 TCP 连接。
  5. 双方使用 `tokio::io::copy_bidirectional` 进行零成本的异步双向字节流拷贝。

### 3.2 动态网络穿透选路 (ICE Routing Module)
针对国内网络，引擎执行高度定制化的并发探测（基于优先级的回退策略）：

1. **IPv6 Direct (最高优)**：收集本地公网 IPv6，尝试绕过 NAT 直接握手。
2. **Local LAN (次高优)**：收集同一路由器下的 `192.168.x.x` 局域网地址。
3. **UDP Hole Punching**：通过 STUN 获取公网 IP:Port，执行标准的对称型/圆锥型 NAT 穿透。
4. **TCP Hole Punching (防 QoS 降级)**：若 UDP 探测丢包严重，自动降级为 TCP 同步打洞（利用 TCP 的 `SO_REUSEADDR` 特性）。
5. **TURN Relay (保底底线)**：若双方皆为严格对称 NAT，立即切换至指定的私有化 TURN 中继节点。

## 4. 内部状态机设计 (State Machine)

引擎内部维护一个严格的状态机，并通过 Stdout 实时推送给启动器，以便 UI 做出响应。

```rust
pub enum SessionState {
    /// 初始状态
    Idle,
    /// 正在连接信令服务器
    ConnectingSignaling,
    /// 房主：房间已创建，等待加入 (附带生成的 Base64 邀请码)
    RoomCreated { invite_code: String },
    /// 客户端：正在解析邀请码并执行 ICE 打洞探测
    PunchingHoles,
    /// 核心状态：P2P 隧道建立成功，TCP 转发已就绪
    TunnelReady { 
        local_proxy_port: u16,   // 客户端启动器应连接此端口
        route_method: RouteType, // 当前使用的路线 (IPv6/UDP/Relay)
        latency_ms: u32          // 实时延迟
    },
    /// 连接中断或异常
    Disconnected(String),
}

pub enum RouteType {
    IPv6Direct,
    LocalNetwork,
    UdpP2p,
    TcpP2p,
    TurnRelay,
}
```

## 5. 进程间通信协议 (IPC: Launcher <-> Sidecar)

引擎不占用任何本地网络端口进行控制流通信，而是通过 **Stdio (Standard I/O) 结合 JSON-RPC 2.0 风格** 与父进程（启动器）对话。所有交互遵循按行分割（Line-delimited JSON）的原则。

### 5.1 指令：房主创建房间 (CREATE_ROOM)
**启动器发送 (Stdin 写入, 以 `\n` 结尾):**
```json
{
  "req_id": "req-001",
  "action": "CREATE_ROOM",
  "payload": {
    "mc_server_port": 25565,
    "signaling_server": "wss://signal.yourdomain.com",
    "turn_server": "turn:turn.yourdomain.com:3478"
  }
}
```
**引擎返回 (Stdout 输出):**
```json
{
  "type": "response",
  "req_id": "req-001",
  "status": "success",
  "data": {
    "invite_code": "eyJzZHAiOiJ2PTB..."
  }
}
```

### 5.2 指令：玩家加入房间 (JOIN_ROOM)
**启动器发送 (Stdin 写入):**
```json
{
  "req_id": "req-002",
  "action": "JOIN_ROOM",
  "payload": {
    "invite_code": "eyJzZHAiOiJ2PTB...",
    "desired_local_port": 50001
  }
}
```
**引擎异步事件推送 (Stdout 输出, Event: TunnelReady):**
由于打洞需要时间，引擎会在成功后主动向控制台打印事件行。
```json
{
  "type": "event",
  "event_name": "STATE_UPDATE",
  "data": {
    "state": "TunnelReady",
    "local_proxy_port": 50001,
    "route_method": "UdpP2p",
    "latency_ms": 32
  }
}
```

### 5.3 事件：房主端玩家状态变动 (PLAYER_EVENT)
当有玩家连接或断开时，房主端的引擎会向 Stdout 推送此事件。
```json
{
  "type": "event",
  "event_name": "PLAYER_JOINED",
  "data": {
    "peer_id": "client-uuid-8899",
    "route_method": "IPv6Direct"
  }
}
```

## 6. 合规与私有化部署配套 (Deployment)

为了使该核心模块在各大启动器中安全落地，需提供以下配套设施：

1. **Signaling/TURN Docker 镜像**：提供一个包含 `webrtc-signal` 和 `coturn` 的轻量级 Docker-compose 文件。官方文档指引：“若公共节点拥堵，请执行 `docker-compose up -d` 私有化部署专属节点。”
2. **流量隔离设计**：`NetCore` 在代码层面写死，只接受来自配置的 Signaling Server 匹配到的 PeerID 的流量，彻底切断被用于构建非法 PCDN 网络或公共 VPN 的可能性。
3. **内存级会话**：信令服务器不挂载任何数据库（Redis/MySQL）。所有的 PeerID 和邀请码仅存在于内存中，进程重启或房间关闭即刻销毁，实现“阅后即焚”，极大降低合规风险。

## 7. 本地安全与进程隔离 (Local Security)

得益于 Sidecar 架构的设计，本引擎在本地安全防范上具备天然的免疫力。

### 7.1 物理隔离级的 IPC 安全
由于放弃了 WebSocket 和 HTTP 本地监听，改用 OS 级别的父子进程管道（Stdin/Stdout）通信，任何外部网页脚本（如 CSRF）或局域网内的恶意软件，**在物理上绝对无法触碰到引擎的控制端点**。唯有拉起该进程的启动器本身拥有通信权限。

### 7.2 TCP 代理端口的“按需闪现”
作为联机隧道入口的本地伪装端口，执行最严格的生命周期管理：
* **严格绑定**：必须死死绑定在 `127.0.0.1`（或 `::1`），绝不绑定 `0.0.0.0`。
* **极度延后与极速销毁**：仅在 P2P 隧道就绪的那一毫秒开始监听；一旦检测到隧道断开、收到关闭指令、或父进程崩溃导致 stdin 管道断裂，TCP 监听器随 Tokio 运行时立即销毁，零安全敞口。

## 8. 异常处理与网络自愈 (Resilience & Recovery)

国内玩家经常遇到 Wi-Fi 信号不稳、路由器重启或突然切换到手机热点的情况，引擎具备“优雅自愈”能力。

### 8.1 隧道保活、心跳与进程兜底
* **网络层**：在 WebRTC DataChannel 内部，引擎每 5 秒发送 Ping 包。连续 3 次未收到 Pong，判定链路假死，推送 `CONNECTION_LOST` 事件。
* **进程层**：引擎的内部循环高度依赖 Stdin 通道。一旦启动器意外崩溃（未发送退出指令），OS 会自动切断管道，引擎立刻触发自毁逻辑，确保不会成为后台“僵尸进程”。

### 8.2 无缝 ICE 重启 (ICE Restart)
网络切换自愈：当玩家从 Wi-Fi 切换到手机热点，引擎无需玩家重新输入邀请码。引擎内部自动触发 ICE Restart 流程，重新收集新 IP 并默默与房主重连。期间本地 MC TCP 代理不主动切断，仅表现为游戏内数秒卡顿。

## 9. 可观测性与日志系统 (Observability)

为方便排查网络问题，引擎采用标准化流式输出。

### 9.1 结构化 JSON 日志流
引擎不写本地日志文件。所有的日志信息会被包装成 JSON 格式打印到 Stdout（使用 `"type": "log"` 字段与 IPC 事件区分），供启动器收集或直接展现在控制台。

```json
{
  "type": "log",
  "level": "WARN",
  "timestamp": "2026-03-30T10:00:00Z",
  "message": "UDP 穿透丢包率过高，正在降级尝试 TCP Hole Punching..."
}
```

### 9.2 严格的隐私脱敏 (Privacy Masking)
日志中的公网 IP 强制执行掩码处理（如：`114.24.xxx.xx`）。玩家交换的 SDP 凭证在打印时必须截断，确保玩家截图求助时不会泄露个人宽带 IP 从而遭到 DDoS 攻击。

## 10. 跨平台编译与分发 (CI/CD & Distribution)

作为极简组件，体积和跨技术栈的兼容性是王道：

### 10.1 现代启动器 Sidecar 完美集成
引擎二进制文件严格遵循无状态（Stateless）设计，不写注册表，不在本地创建配置文件。
* **Tauri / Vite 栈兼容**：完美适配 Tauri 的 `api/process/Command` Sidecar 官方 API。开发者只需在 `tauri.conf.json` 中配置二进制路径，即可通过前端 TS 代码直接监听 stdout 事件流。
* **C# / Java 栈兼容**：通过 `System.Diagnostics.Process` 或 `ProcessBuilder` 拉起可执行文件，并重定向标准输入/输出流即可无缝对接。

### 10.2 极限体积与静态链接
* **Windows**：采用 `x86_64-pc-windows-gnu` 目标或静态链接 MSVC，确保在纯净系统上免环境双击即跑。
* **Linux/macOS**：使用 `musl` 替代 `glibc` 进行完全静态编译，保证在任何发行版中完美兼容。
* 开启 `lto = true` 和 `strip = true` 编译选项，将单文件核心引擎体积压缩在 5MB - 10MB，零负担集成入各大主流启动器。
