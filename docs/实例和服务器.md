# Minecraft 启动器数据库设计文档 (多实例-多服务器架构)

## 1. 架构概述
本设计采用**多对多（Many-to-Many）**的关系型架构，将“游戏实例（Instance）”与“服务器资源库（Server Pool）”解耦。
* **高复用性**：同一个服务器（如 Hypixel）只需在数据库中存储一次，即可分配给多个不同的游戏实例。
* **高扩展性**：支持为单个实例配置完整的“多人游戏”列表预设，包括排序、主服务器设置等。

---

## 2. 实体关系模型 (ERD)

* **`instances` (1) <---> (N) `instance_servers` (N) <---> (1) `servers`**
* 一个实例可以包含多个服务器。
* 一个服务器可以被多个实例引用。
* `instance_servers` 作为中间表，不仅负责关联，还承载了该服务器在特定实例中的个性化设置（如排序、是否自动连接）。

---

## 3. 数据表详细设计

### 3.1 实例主表 (`instances`)
负责存储游戏实例的核心运行环境配置。

| 字段名 | 数据类型 | 键/约束 | 默认值 | 说明 |
| :--- | :--- | :--- | :--- | :--- |
| `id` | VARCHAR(36) | **PK** | UUID | 实例唯一标识符 |
| `name` | VARCHAR(255) | Not Null | | 实例名称 (例："1.20.1 极限生存包") |
| `mc_version` | VARCHAR(50) | Not Null | | Minecraft 核心版本 (例："1.20.1") |
| `loader_type` | VARCHAR(50) | Nullable | 'Vanilla' | 隔离/加载器类型 (Vanilla/Forge/Fabric/NeoForge等) |
| `loader_version` | VARCHAR(50) | Nullable | | 加载器版本号 |
| `java_path` | VARCHAR(500) | Nullable | NULL | 自定义 Java 路径 (为空则读取启动器全局配置) |
| `min_memory` | INT | Nullable | 1024 | 最小分配内存 (MB) |
| `max_memory` | INT | Nullable | 4096 | 最大分配内存 (MB) |
| `icon_path` | VARCHAR(255) | Nullable | | 启动器UI中该实例的图标路径 |
| `created_at` | DATETIME | | CURRENT_TIMESTAMP| 实例创建时间 |
| `updated_at` | DATETIME | | CURRENT_TIMESTAMP| 实例最后修改时间 |

### 3.2 服务器资源库表 (`servers`)
全局服务器地址簿，统一管理所有已知的服务器 IP 信息。

| 字段名 | 数据类型 | 键/约束 | 默认值 | 说明 |
| :--- | :--- | :--- | :--- | :--- |
| `id` | VARCHAR(36) | **PK** | UUID | 服务器唯一标识符 |
| `name` | VARCHAR(255) | Not Null | | 服务器展示名称 (例："Hypixel Network") |
| `address` | VARCHAR(255) | Not Null | | 服务器 IP 或 域名 (例："mc.hypixel.net") |
| `port` | INT | Not Null | 25565 | 端口号 (1-65535) |
| `icon_base64` | TEXT | Nullable | NULL | Base64编码的服务器图标 (直接对应 `servers.dat` 规范) |
| `hide_address` | BOOLEAN | Not Null | FALSE | 是否在客户端隐藏真实 IP 地址 |

### 3.3 实例-服务器关联表 (`instance_servers`)
维护实例与服务器列表的映射关系及个性化状态。

| 字段名 | 数据类型 | 键/约束 | 默认值 | 说明 |
| :--- | :--- | :--- | :--- | :--- |
| `instance_id` | VARCHAR(36) | **FK**, **PK** | | 关联 `instances.id` |
| `server_id` | VARCHAR(36) | **FK**, **PK** | | 关联 `servers.id` |
| `is_primary` | BOOLEAN | Not Null | FALSE | 是否为**主服务器**（启动后通过参数 `--server` 直连）|
| `sort_order` | INT | Not Null | 0 | 决定多人游戏列表中的排序权重 (数字越小越靠前) |
| `added_at` | DATETIME | | CURRENT_TIMESTAMP| 添加到该实例的时间 |

> **注**：`instance_id` 和 `server_id` 共同组成复合主键（Composite Primary Key），确保同一个实例下不会重复添加同一个服务器。

---

## 4. 索引优化建议 (Indexes)

为了保证启动器 UI 的响应速度，建议在数据库初始化时创建以下索引：
1.  **`idx_instance_servers_instance`**: 在 `instance_servers(instance_id, sort_order)` 上建立复合索引。当启动器准备生成某个实例的服务器列表时，能够极快地拉取并按顺序排列。
2.  **`idx_servers_address`**: 在 `servers(address, port)` 上建立索引。当用户手动输入一个新 IP 时，可以快速查重，避免在全局库中产生重复的服务器条目。

---

## 5. 核心业务逻辑与数据流 (Data Flow)

作为设计者，你需要关注数据如何从数据库“变成”Minecraft 能识别的文件。

### 场景一：生成 `servers.dat` (预置多人游戏列表)
Minecraft 本体并不读取关系型数据库，它只认实例根目录下的 `servers.dat`（这是一个 NBT 格式的二进制文件）。
* **触发时机**：玩家点击“启动游戏”的瞬间。
* **执行逻辑**：
    1.  执行 SQL 查询获取该实例关联的所有服务器：
        ```sql
        SELECT s.name, s.address, s.port, s.icon_base64, s.hide_address
        FROM servers s
        JOIN instance_servers is_ ON s.id = is_.server_id
        WHERE is_.instance_id = '目标实例UUID'
        ORDER BY is_.sort_order ASC;
        ```
    2.  启动器（C#/Java/C++/Rust）将查询结果序列化为 NBT 结构。
    3.  静默覆盖或写入到 `<实例路径>/.minecraft/servers.dat` 中。

### 场景二：一键直连 (Auto-Connect)
* **触发时机**：玩家点击“启动游戏”。
* **执行逻辑**：
    1.  查询该实例是否有被标记为 `is_primary = TRUE` 的服务器。
        ```sql
        SELECT s.address, s.port
        FROM servers s
        JOIN instance_servers is_ ON s.id = is_.server_id
        WHERE is_.instance_id = '目标实例UUID' AND is_.is_primary = 1
        LIMIT 1;
        ```
    2.  如果查询有结果，在拼接 Minecraft 启动参数时，加入：`--server <address> --port <port>`。游戏加载完毕后将跳过主菜单，直接进入该服务器。
