# 开发模式启动性能分析报告与优化建议

## 1. 现状描述
当前项目在执行 `npm run tauri dev` 启动时，用户反馈从执行命令到窗口出现，以及窗口出现到界面完全可交互的等待时间过长。

## 2. 核心问题分析

### 2.1 Rust 后端初始化阻塞
*   **同步数据库加载**：在 `lib.rs` 中，使用 `block_on` 方式初始化 SQLite 数据库。这会阻塞 Tauri 主线程，导致窗口创建必须等待数据库迁移和连接池建立。
*   **Setup 钩子负载**：在 `setup` 周期内执行了身份生成、局域网广播启动等同步或密集型任务。

### 2.2 前端启动逻辑负载 (App.tsx)
*   **高开销异步任务**：`initGamepadModRegistry` 在启动时从网络拉取全量手柄 Mod 版本库，涉及大量 API 请求和 JSON 数据处理。
*   **I/O 密集型操作**：`triggerJavaAutoDetect` 在启动时进行全量或关键路径的 Java 环境扫描，造成磁盘 I/O 争用。
*   **级联模块请求**：过度使用 `lazy` 加载配合多个 `Suspense` 块，导致 Vite 在开发模式下需要进行多轮按需编译，增加了“首屏时间”。

### 2.3 开发环境配置 (Vite)
*   **依赖预构建开销**：`three.js`、`skinview3d` 等大型依赖库在初次启动时需要 Vite 进行繁重的预处理。
*   **碎文件加载**：大量的 CSS 文件独立导入和图标库模块，导致浏览器在开发阶段产生数百个 HTTP 请求。

---

## 3. 优化建议

### 3.1 后端优化
*   **[建议] 异步数据库初始化**：移除 `block_on`，改用异步方式初始化数据库，或将数据库加载移至窗口显示之后。
*   **[建议] 延迟服务启动**：将 mDNS 广播和手柄监听等非核心服务延迟到窗口完全显示后再执行。

### 3.2 前端逻辑优化
*   **[建议] 任务分片与延迟执行**：
    *   将 `initGamepadModRegistry` 延迟到用户进入相关页面（如“安装实例”或“实例详情”）时再触发，或转为后台闲暇时（Idle）执行。
    *   Java 检测改为按需触发，或者在启动时仅检查已知路径。
*   **[建议] 减少初次 Suspense 压力**：
    *   将核心页面（如 `Home`）改为普通导入，而非 `lazy` 加载，减少首屏渲染时的请求级联。
    *   合并零碎的 CSS 文件，或利用 Vite 的 CSS 预处理器功能。

### 3.3 构建工具优化
*   **[建议] 优化图标导入**：确保使用解构导入或专门的图标插件，防止 Vite 加载整个 `lucide-react` 包。
*   **[建议] 细化依赖预构建配置**：在 `vite.config.ts` 中显式指定需要预构建的包，减少初次加载时的猜测时间。

---

## 4. 实施优先级
1.  **高**：延迟执行手柄注册表初始化（解决渲染卡顿）。
2.  **高**：解耦 Rust 端的同步数据库阻塞（解决窗口出现慢）。
3.  **中**：优化 `Home` 页面加载策略（解决白屏等待）。
4.  **低**：合并 CSS 资源（持续优化）。
